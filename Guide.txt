Here is the diagnosis and solution for each:

### 1. Why Only One User Shows Up
**The Problem:** Your migration script failed with `403 Forbidden` / `violates row-level security policy`.
*   The script tried to push *all* users from IndexedDB to Supabase.
*   The RLS policy `"Users can insert own profile"` only allows `auth.uid() = id`.
*   Since you were logged in as Damien, you could only upsert Damien's record. Tinashe's record was rejected because your ID didn't match hers.
*   **Result:** Supabase has 1 user. IndexedDB still has 2. The app prefers Supabase when online, so it shows 1.

**The Fix:** You need a temporary "Super Admin" policy or use the Service Role Key (not recommended for client-side). The safest immediate fix is to update the INSERT policy to allow admins to create users in their org.

Run this in Supabase SQL Editor:
```sql
-- Update INSERT policy to allow admins to create users in their organization
DROP POLICY IF EXISTS "Users can insert own profile" ON public.user_profiles;

CREATE POLICY "Users and admins can insert profiles" 
ON public.user_profiles FOR INSERT 
WITH CHECK (
  auth.uid() = id OR 
  (
    (auth.jwt() ->> 'isAdmin')::boolean = true AND
    "organizationName" = (auth.jwt() ->> 'organizationName')::text
  )
);
```

After running this, click **"Migrate & Sync"** again. It should now push Tinashe's record to Supabase without error.

### 2. Why Revoking Admin Rights "Failed Silently"
**The Problem:** The notification says success because the code updates **IndexedDB first**, then tries Supabase.
*   `updateUserRole` in `auth-context.tsx` updates IndexedDB immediately → UI reflects change locally.
*   Then it calls `supabase.from('user_profiles').update(...)`.
*   This likely fails silently due to RLS (similar to the migration issue) or because the JWT metadata hasn't refreshed.
*   When you refresh, `getUsers()` fetches from Supabase, which still has the old `isAdmin: true`, overwriting your local change.

**The Fix:** Ensure the UPDATE policy explicitly allows admins to modify roles. The policy I provided earlier (`"Users can update own or org profiles"`) should handle this, but verify it exists. Also, add error handling to `updateUserRole`:

```typescript
// In auth-context.tsx, updateUserRole function
const updateUserRole = async (userId: string, role: string) => {
  await userDB.updateUser(userId, { role: role as any, isAdmin: role === 'admin' });
  
  if (navigator.onLine) {
    const { error } = await supabase
      .from('user_profiles')
      .update({ role: role, isAdmin: role === 'admin' })
      .eq('id', userId);
      
    if (error) {
      console.error('[AuthContext] Role update failed on Supabase:', error);
      // Revert local change if remote fails
      await userDB.updateUser(userId, { role: role === 'admin' ? 'user' : 'admin', isAdmin: role !== 'admin' });
      throw new Error(`Failed to sync role change: ${error.message}`);
    }
  }
  return { success: true };
};
```

### 3. Why System Setup Button Still Appears
**The Problem:** The setup check in `page.tsx` queries Supabase directly:
```typescript
const { data } = await supabase.from('user_profiles').select('isAdmin').eq('isAdmin', true).limit(1);
```
*   Since the migration failed for Tinashe, Supabase might only have Damien (or neither if sync is broken).
*   If Supabase returns 0 admins, the button appears.
*   Additionally, this query lacks an organization filter, so it's checking the *entire* database.

**The Fix:** Update the setup check to also consult IndexedDB as a fallback, and filter by organization:

```typescript
// In page.tsx, checkInitialState function
const checkInitialState = async () => {
  try {
    const currentOrgSlug = slugifyOrg("ACRUX IT SOLUTIONS"); // Or get from context
    
    // Check IndexedDB FIRST (source of truth when sync is broken)
    const localUsers = await userDB.getAllUsers();
    const hasLocalAdmin = localUsers.some(u => u.isAdmin === true && slugifyOrg(u.organizationName) === currentOrgSlug);
    
    let hasRemoteAdmin = false;
    if (navigator.onLine && !hasLocalAdmin) { // Only check remote if local is empty
      try {
        const { data } = await supabase
          .from('user_profiles')
          .select('isAdmin')
          .eq('isAdmin', true)
          .eq('organizationName', currentOrgSlug) // Add org filter!
          .limit(1);
        hasRemoteAdmin = data && data.length > 0;
      } catch (e) {
        console.error('Error checking Supabase for admin:', e);
      }
    }
    
    setShowSetupGuide(!hasLocalAdmin && !hasRemoteAdmin);
  } catch (error) {
    console.error('Error checking users:', error);
  }
};
```
